# 第5章 系统实现

本章展示各功能模块的关键实现代码和界面效果。代码均从实际项目中提取，结合核心业务逻辑进行详细分析。

## 5.1 用户认证模块实现

用户认证模块是整个系统的安全基础，涵盖用户注册、登录、JWT令牌管理以及统一响应封装等功能。本模块采用前后端分离架构下的无状态认证方案，通过JWT（JSON Web Token）实现用户身份的持久化验证。

### 5.1.1 用户注册实现

用户注册功能由`UserServiceImpl`类中的`register`方法实现。该方法负责接收前端传入的用户信息，经过一系列校验和处理后将用户数据持久化到数据库。以下为注册方法的核心代码：

```java
@Override
public void register(User user) {
    LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(User::getUsername, user.getUsername());
    if (userMapper.selectOne(wrapper) != null) {
        throw new BusinessException("用户名已存在");
    }
    user.setPassword(SecureUtil.md5(user.getPassword() + SALT));
    if (user.getRole() == null) {
        user.setRole(0);
    }
    if (user.getStatus() == null) {
        user.setStatus(1);
    }
    userMapper.insert(user);
}
```

注册方法的执行流程如下：首先，通过MyBatis-Plus提供的`LambdaQueryWrapper`构建查询条件，根据用户名查询数据库中是否已存在同名用户。若查询结果不为空，则抛出`BusinessException`业务异常，提示"用户名已存在"。通过用户名唯一性校验后，系统利用Hutool工具库中的`SecureUtil.md5()`方法对用户密码进行MD5加盐加密处理。其中，盐值`SALT`定义为常量字符串`"pet_kingdom_salt"`，将用户原始密码与盐值拼接后进行MD5摘要运算，有效防止彩虹表攻击和撞库破解。加密完成后，系统为用户设置默认角色（`role=0`表示普通用户）和默认状态（`status=1`表示正常），最终通过MyBatis-Plus的`insert`方法将用户实体写入数据库。

### 5.1.2 用户登录实现

用户登录功能由`UserServiceImpl`类中的`login`方法实现。该方法负责验证用户身份凭证，验证通过后生成JWT令牌并返回给前端。以下为登录方法的核心代码：

```java
@Override
public Map<String, Object> login(String username, String password) {
    LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(User::getUsername, username);
    User user = userMapper.selectOne(wrapper);
    if (user == null) {
        throw new BusinessException("用户名或密码错误");
    }
    if (!SecureUtil.md5(password + SALT).equals(user.getPassword())) {
        throw new BusinessException("用户名或密码错误");
    }
    if (user.getStatus() == 0) {
        throw new BusinessException("账号已被禁用");
    }
    String token = jwtUtil.generateToken(user.getId(), user.getUsername(), user.getRole());
    user.setPassword(null);
    Map<String, Object> result = new HashMap<>();
    result.put("token", token);
    result.put("user", user);
    return result;
}
```

登录方法的执行流程分为四个关键步骤。第一步，根据用户名查询数据库获取用户实体，若用户不存在则返回模糊错误提示"用户名或密码错误"，避免泄露用户是否存在的信息。第二步，将前端传入的密码以相同的MD5加盐方式进行加密，并与数据库中存储的密码密文进行比对，不一致时返回相同的模糊错误提示。第三步，检查用户账号状态，当`status`为0时表示账号已被管理员禁用，拒绝登录。第四步，上述验证全部通过后，调用`JwtUtil.generateToken()`方法生成JWT令牌，将令牌和脱敏后的用户信息（密码字段置空）封装到`Map`中返回给前端。这种设计将密码密文从响应数据中移除，确保敏感信息不会通过网络传输泄露。

### 5.1.3 JWT工具类实现

JWT工具类`JwtUtil`封装了令牌的生成与验证逻辑，是系统无状态认证机制的核心组件。该类基于`auth0-java-jwt`库实现，以下为关键方法代码：

```java
@Component
public class JwtUtil {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.expiration}")
    private long expiration;

    public String generateToken(Long userId, String username, Integer role) {
        Date now = new Date();
        Date expireDate = new Date(now.getTime() + expiration);
        return JWT.create()
                .withClaim("userId", userId)
                .withClaim("username", username)
                .withClaim("role", role)
                .withIssuedAt(now)
                .withExpiresAt(expireDate)
                .sign(Algorithm.HMAC256(secret));
    }

    public boolean verify(String token) {
        try {
            JWTVerifier verifier = JWT.require(Algorithm.HMAC256(secret)).build();
            verifier.verify(token);
            return true;
        } catch (TokenExpiredException e) {
            return false;
        } catch (JWTDecodeException e) {
            return false;
        }
    }
}
```

`generateToken`方法接收用户ID、用户名和角色三个参数作为令牌的载荷（Payload），通过`JWT.create()`构建器模式创建令牌。令牌中包含三个自定义声明（Claim）：`userId`用于后续请求中识别用户身份，`username`用于展示用户信息，`role`用于权限判断。同时设置签发时间（`issuedAt`）和过期时间（`expiresAt`），过期时间由配置文件中的`jwt.expiration`参数控制。最终使用HMAC-SHA256算法和密钥对令牌进行签名，确保令牌的完整性和不可篡改性。

`verify`方法使用相同的密钥和算法构建验证器，对传入的令牌进行验证。若令牌有效且未过期，返回`true`；若令牌已过期（`TokenExpiredException`）或格式错误（`JWTDecodeException`），则返回`false`。此外，该类还提供了`getUserId`、`getUsername`、`getRole`三个方法，通过解码令牌获取对应的声明值，供拦截器使用。

### 5.1.4 JWT拦截器实现

JWT拦截器`JwtInterceptor`实现了Spring MVC的`HandlerInterceptor`接口，在请求到达控制器之前对JWT令牌进行统一校验。以下为拦截器的`preHandle`方法代码：

```java
@Override
public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
    if ("OPTIONS".equalsIgnoreCase(request.getMethod())) {
        return true;
    }

    String token = request.getHeader("Authorization");
    if (StringUtils.isBlank(token)) {
        token = request.getParameter("token");
    }

    if (StringUtils.isBlank(token)) {
        throw new BusinessException(401, "未登录，请先登录");
    }

    if (!token.startsWith("Bearer ")) {
        throw new BusinessException(401, "Token格式错误");
    }

    token = token.substring(7);
    if (!jwtUtil.verify(token)) {
        throw new BusinessException(401, "Token已过期或无效，请重新登录");
    }

    request.setAttribute("userId", jwtUtil.getUserId(token));
    request.setAttribute("username", jwtUtil.getUsername(token));
    request.setAttribute("role", jwtUtil.getRole(token));
    return true;
}
```

拦截器的执行逻辑分为五个阶段。首先，对浏览器的预检请求（OPTIONS方法）直接放行，这是处理跨域请求时的标准做法。其次，从请求头的`Authorization`字段获取令牌，若该字段为空则尝试从请求参数中获取。第三步，若仍未获取到令牌，抛出401状态码的业务异常，提示用户未登录。第四步，验证令牌格式是否以`Bearer `前缀开头，符合OAuth 2.0标准的Bearer Token规范，并截取实际令牌字符串。第五步，调用`JwtUtil.verify()`验证令牌的有效性，验证通过后将用户ID、用户名和角色信息注入到请求属性中，供后续控制器方法通过`request.getAttribute()`获取。这种设计使得控制器无需关心认证细节，只需从请求属性中获取当前用户信息即可。

### 5.1.5 统一响应封装

系统采用统一的响应格式封装所有API的返回结果，确保前端能够以一致的方式处理响应数据。该封装由`Result`泛型类实现，代码如下：

```java
@Data
public class Result<T> {

    private int code;
    private String message;
    private T data;

    private Result() {}

    public static <T> Result<T> success() {
        Result<T> result = new Result<>();
        result.setCode(200);
        result.setMessage("success");
        return result;
    }

    public static <T> Result<T> success(T data) {
        Result<T> result = new Result<>();
        result.setCode(200);
        result.setMessage("success");
        result.setData(data);
        return result;
    }

    public static <T> Result<T> error(String message) {
        Result<T> result = new Result<>();
        result.setCode(500);
        result.setMessage(message);
        return result;
    }

    public static <T> Result<T> error(int code, String message) {
        Result<T> result = new Result<>();
        result.setCode(code);
        result.setMessage(message);
        return result;
    }
}
```

`Result`类包含三个字段：`code`表示响应状态码（200表示成功，500表示服务器错误，401表示未认证等），`message`表示响应消息，`data`为泛型字段承载实际的业务数据。该类通过私有构造函数和静态工厂方法的设计模式，确保只能通过`success()`或`error()`方法创建实例。成功响应提供两个重载方法：无参版本用于不需要返回数据的操作（如删除、更新），带参版本用于返回业务数据。错误响应同样提供两个重载方法：仅包含消息的版本默认使用500状态码，带状态码的版本用于特定错误场景（如401未认证、403无权限等）。所有控制器方法均返回`Result`类型，实现了前后端数据交互格式的标准化。

## 5.2 宠物信息管理模块实现

宠物信息管理模块是萌宠乐园平台的核心业务模块之一，实现宠物信息的增删改查、分类管理等功能。该模块基于MyBatis-Plus的`LambdaQueryWrapper`构建类型安全的查询条件，结合分页插件实现高效的数据检索。

### 5.2.1 宠物列表查询实现

宠物列表查询由`PetController`中的`list`方法提供接口入口，代码如下：

```java
@GetMapping("/list")
public Result<Page<Pet>> list(@RequestParam(defaultValue = "1") int page,
                              @RequestParam(defaultValue = "10") int size,
                              @RequestParam(required = false) Long categoryId,
                              @RequestParam(required = false) String keyword) {
    return Result.success(petService.listPets(page, size, categoryId, keyword));

```

该接口支持四个查询参数：`page`为当前页码（默认第1页），`size`为每页记录数（默认10条），`categoryId`为宠物分类ID（可选，用于按分类筛选），`keyword`为搜索关键词（可选，用于模糊搜索）。控制器通过统一响应封装`Result.success()`将分页结果返回给前端。

在服务层，分页查询通过MyBatis-Plus的`LambdaQueryWrapper`构建动态查询条件。当`categoryId`不为空时，添加等值查询条件过滤指定分类的宠物；当`keyword`不为空时，添加`LIKE`模糊查询条件在宠物名称和品种字段中搜索匹配的记录。查询结果按创建时间降序排列，确保最新添加的宠物优先展示。分页功能由MyBatis-Plus的`Page`对象实现，自动执行`COUNT`查询获取总记录数，并在SQL语句中追加`LIMIT`和`OFFSET`子句实现物理分页，避免内存中处理大量数据带来的性能问题。

### 5.2.2 宠物分类管理实现

宠物分类管理采用标准的CRUD（创建、读取、更新、删除）实现模式。分类数据的持久化操作基于MyBatis-Plus的`BaseMapper`接口提供的通用方法：新增分类使用`insert`方法，查询全部分类使用`selectList`方法，更新分类使用`updateById`方法，删除分类使用`deleteById`方法。分类实体包含分类名称、排序号、创建时间等字段。在宠物列表页面中，分类数据以侧边栏形式展示，用户点击分类名称后，前端将分类ID作为参数传递给列表查询接口，实现按分类筛选宠物的功能。

## 5.4 商品与订单模块实现

商品与订单模块是平台电子商务功能的核心，实现商品的展示与管理、购物车、订单创建与查询等功能。订单创建过程涉及商品校验、金额计算、库存扣减等复杂业务逻辑，需要保证数据一致性。

### 5.4.1 商品管理实现

商品管理同样采用标准的CRUD实现模式。商品实体包含名称、价格、库存、销量、图片、描述等字段。商品列表支持分页查询和关键词搜索，管理员可以在后台进行商品的添加、编辑和删除操作。商品详情页面展示商品的完整信息，包括图片轮播、价格、库存状态和商品描述，用户可以将商品加入购物车或直接下单。

### 5.4.2 订单创建实现

订单创建是商品与订单模块中最复杂的业务操作，由`OrderServiceImpl`中的`create`方法实现。该方法使用Spring的`@Transactional`注解确保事务一致性。以下为核心代码：

```java
@Override
@Transactional
public Order create(Long userId, List<OrderItem> items, String address, String phone, String remark) {
    if (items == null || items.isEmpty()) {
        throw new BusinessException("订单商品不能为空");
    }

    BigDecimal totalAmount = BigDecimal.ZERO;
    for (OrderItem item : items) {
        Product product = productMapper.selectById(item.getProductId());
        if (product == null) {
            throw new BusinessException("商品不存在: " + item.getProductId());
        }
        if (product.getStock() < item.getQuantity()) {
            throw new BusinessException("商品库存不足: " + product.getName());
        }
        item.setProductName(product.getName());
        item.setProductImage(product.getImage());
        item.setPrice(product.getPrice());

        totalAmount = totalAmount.add(product.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())));

        product.setStock(product.getStock() - item.getQuantity());
        product.setSales(product.getSales() + item.getQuantity());
        productMapper.updateById(product);
    }

    Order order = new Order();
    order.setOrderNo(generateOrderNo());
    order.setUserId(userId);
    order.setTotalAmount(totalAmount);
    order.setStatus(0);
    order.setAddress(address);
    order.setPhone(phone);
    order.setRemark(remark);
    order.setCreateTime(LocalDateTime.now());
    order.setUpdateTime(LocalDateTime.now());
    orderMapper.insert(order);

    for (OrderItem item : items) {
        item.setOrderId(order.getId());
        orderItemMapper.insert(item);
    }

    return order;
}
```

订单创建方法的执行流程包含五个关键步骤。第一步，校验订单商品列表是否为空，为空则直接拒绝创建。第二步，遍历所有订单项（OrderItem），逐一进行商品校验：根据商品ID查询商品是否存在，检查库存是否充足。校验通过后，将商品名称、图片和单价快照到订单项中，确保即使商品信息后续发生变化，历史订单仍能准确反映下单时的商品状态。第三步，计算订单总金额：使用`BigDecimal`类型进行精确的货币计算，将每项商品的单价乘以购买数量后累加到总金额中，避免浮点数运算的精度丢失问题。第四步，生成唯一订单号并创建订单主记录，订单号由前缀"PK"、当前时间戳和四位随机数拼接而成，具有极高的唯一性保证。第五步，将所有订单项关联到订单主记录并逐条插入数据库。

整个方法通过`@Transactional`注解实现声明式事务管理。当任何步骤出现异常（如商品不存在、库存不足），所有已执行的数据库操作将自动回滚，包括已扣减的库存和已增加的销量，确保数据的一致性和完整性。库存扣减采用先查询后更新的方式，在事务保护下避免超卖问题。

## 5.5 社区论坛模块实现

社区论坛模块为用户提供一个交流互动的平台，支持帖子的发布、浏览、搜索以及评论功能。该模块的特点在于需要关联查询用户信息和评论统计数据，以丰富帖子的展示内容。

### 5.5.1 帖子列表查询实现

帖子列表查询由`PostServiceImpl`中的`listPosts`方法实现。该方法不仅查询帖子基本信息，还关联查询发帖用户的昵称和头像，并统计每篇帖子的评论数量。以下为核心代码：

```java
@Override
public IPage<PostVO> listPosts(int page, int size, String keyword) {
    Page<Post> postPage = postMapper.selectPage(pageParam, wrapper);

    List<Long> userIds = postPage.getRecords().stream()
            .map(Post::getUserId).distinct().collect(Collectors.toList());
    Map<Long, User> userMap = userIds.isEmpty() ? Collections.emptyMap() :
            userMapper.selectBatchIds(userIds).stream()
                    .collect(Collectors.toMap(User::getId, u -> u));

    List<Long> postIds = postPage.getRecords().stream()
            .map(Post::getId).collect(Collectors.toList());
    Map<Long, Integer> commentCountMap = postIds.isEmpty() ? Collections.emptyMap() :
            postIds.stream().collect(Collectors.toMap(
                pid -> pid,
                pid -> Math.toIntExact(commentMapper.selectCount(
                        new LambdaQueryWrapper<Comment>().eq(Comment::getPostId, pid)))
            ));

    IPage<PostVO> voPage = postPage.convert(post -> {
        PostVO vo = new PostVO();
        vo.setId(post.getId());
        vo.setTitle(post.getTitle());
        vo.setContent(post.getContent());
        User user = userMap.get(post.getUserId());
        if (user != null) {
            vo.setUsername(user.getUsername());
            vo.setAvatar(user.getAvatar());
        }
        vo.setCommentCount(commentCountMap.getOrDefault(post.getId(), 0));
        return vo;
    });

    return voPage;
}
```

该方法采用批量查询优化策略，避免N+1查询问题。具体实现分为三个阶段：首先，执行帖子的分页查询，查询条件包括非隐藏帖子的过滤和关键词的模糊匹配，结果按置顶优先、时间降序排列。其次，从分页结果中提取所有不重复的用户ID和帖子ID，分别通过`selectBatchIds`批量查询用户信息和逐条统计评论数量，将结果映射为以ID为键的Map结构以便快速查找。最后，利用MyBatis-Plus分页对象的`convert`方法，将`Post`实体转换为`PostVO`视图对象，在转换过程中从Map中查找并填充用户信息和评论数量。这种批量查询加Map映射的方式，将原本需要N+1次数据库查询的操作优化为固定次数（帖子查询+用户批量查询+评论数统计），显著提升了列表页面的加载性能。

### 5.6.1 数据统计实现

数据统计功能由`StatsController`中的`dashboard`方法实现，为管理员提供系统运营的核心指标数据。代码如下：

```java
@GetMapping("/dashboard")
public Result<Map<String, Object>> dashboard() {
    Map<String, Object> stats = new HashMap<>();
    stats.put("userCount", userMapper.selectCount(null));
    stats.put("petCount", petMapper.selectCount(null));
    stats.put("orderCount", orderMapper.selectCount(null));
    stats.put("appointmentCount", appointmentMapper.selectCount(null));
    stats.put("postCount", postMapper.selectCount(null));

    LocalDateTime todayStart = LocalDate.now().atStartOfDay();
    LocalDateTime todayEnd = LocalDate.now().atTime(LocalTime.MAX);
    stats.put("todayOrderCount", orderMapper.selectCount(
            new LambdaQueryWrapper<Order>()
                    .between(Order::getCreateTime, todayStart, todayEnd)
    ));

    return Result.success(stats);
}
```

该方法通过注入各实体的Mapper接口，调用MyBatis-Plus提供的`selectCount(null)`方法（参数为null表示无条件统计）获取用户总数、宠物总数、订单总数、预约总数和帖子总数五个核心统计指标。此外，通过构建时间范围查询条件，统计当日新增订单数量：以当天零点为起始时间、当天最后一刻（23:59:59）为结束时间，使用`between`条件查询创建时间在该范围内的订单数量。所有统计数据封装到一个Map中返回，前端仪表盘页面根据这些数据渲染统计卡片和图表。

## 5.7 前端页面实现

前端基于Vue.js 2.x框架和Element UI组件库构建，采用Vuex进行状态管理、Vue Router进行路由控制、Axios进行HTTP请求。本节展示前端项目的关键配置和核心实现代码。

### 5.7.1 前端项目配置

前端项目的开发环境配置在`vue.config.js`中完成，核心配置如下：

```javascript
const { defineConfig } = require('@vue/cli-service')
module.exports = defineConfig({

  devServer: {
    port: 8080,
    proxy: {
      '/api': {

  }

```

配置中指定前端开发服务器运行在8080端口，通过`proxy`代理配置解决开发环境下的跨域问题。当浏览器请求以`/api`开头的接口时，开发服务器自动将请求转发到后端服务`http://localhost:8088`，并设置`changeOrigin: true`修改请求头中的Origin字段为目标地址，从而绕过浏览器的同源策略限制。在生产环境中，通过Nginx反向代理实现同样的效果。

### 5.7.2 Axios请求封装

HTTP请求库Axios通过拦截器机制实现了Token自动注入和错误统一处理。以下为`request.js`的关键代码：

```javascript
const service = axios.create({
  baseURL: 'http://localhost:8088/api',
  timeout: 10000
})

service.interceptors.request.use(
  config => {
    const token = localStorage.getItem('token')
    if (token) {
      config.headers['Authorization'] = 'Bearer ' + token
    }
    return config
  },
  error => {
    return Promise.reject(error)
  }
)

service.interceptors.response.use(
  response => {
    const res = response.data
    if (res.code && res.code !== 200) {
      Message({
        message: res.message || '请求失败',
        type: 'error',
        duration: 3000
      })
      return Promise.reject(new Error(res.message || '请求失败'))
    }
    return res
  },
  error => {
    if (error.response) {
      const status = error.response.status
      if (status === 401) {
        Message({
          message: '登录已过期，请重新登录',
          type: 'warning',
          duration: 2000
        })
        localStorage.removeItem('token')
        localStorage.removeItem('userInfo')
        router.push('/login')
      } else if (status === 403) {
        Message({ message: '没有权限访问该资源', type: 'error', duration: 3000 })
      } else if (status === 500) {
        Message({ message: '服务器内部错误', type: 'error', duration: 3000 })
      }
    }
    return Promise.reject(error)
  }
)
```

请求拦截器的核心功能是从`localStorage`中读取JWT令牌，若令牌存在则自动添加到请求头的`Authorization`字段中，遵循Bearer Token规范格式。响应拦截器实现了两层错误处理机制：第一层检查业务状态码，当后端返回的`code`字段不为200时，通过Element UI的`Message`组件弹出错误提示；第二层处理HTTP状态码异常，针对401（未认证）自动清除本地存储的令牌和用户信息并跳转到登录页，针对403（无权限）和500（服务器错误）分别给出对应的提示信息。此外，文件还封装了`get`、`post`、`put`、`del`四个便捷方法，简化各业务模块的接口调用。

### 5.7.3 路由配置

前端路由通过Vue Router实现，包含前台路由和后台路由两组配置，并通过全局前置守卫实现登录拦截和权限控制。路由守卫代码如下：

```javascript
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  if (to.path.startsWith('/admin') && !token) {
    next('/login')
  } else if ((to.path === '/login' || to.path === '/register') && token) {
    next('/')
  } else {
    next()
  }
})
```

路由守卫实现了两条核心规则：第一，当用户访问以`/admin`开头的后台管理页面时，若本地不存在JWT令牌，则自动重定向到登录页面，防止未授权用户直接通过URL访问后台功能。第二，当已登录用户访问登录或注册页面时，自动重定向到首页，避免已认证用户重复登录。前台页面（宠物浏览、社区论坛等）对登录状态不做强制要求，允许未登录用户浏览公开内容，仅在需要用户身份的操作（如发帖、下单、预约）时才要求登录。

路由配置采用懒加载方式（动态`import()`语法），每个页面组件在首次访问时才加载对应的JavaScript文件，有效减小初始包体积，提升首屏加载速度。前台路由挂载在`FrontLayout`布局组件下，后台路由挂载在`AdminLayout`布局组件下，形成清晰的页面层级结构。

### 5.7.4 Vuex状态管理

用户状态管理通过Vuex模块化机制实现，`user.js`模块负责管理用户的登录状态、令牌和个人信息。以下为关键代码：

```javascript
const state = {
  token: localStorage.getItem('token') || '',
  userInfo: JSON.parse(localStorage.getItem('userInfo') || '{}')
}

const mutations = {
  SET_TOKEN(state, token) {
    state.token = token
    localStorage.setItem('token', token)
  },
  SET_USER_INFO(state, userInfo) {
    state.userInfo = userInfo
    localStorage.setItem('userInfo', JSON.stringify(userInfo))
  },
  CLEAR_USER(state) {
    state.token = ''
    state.userInfo = {}
    localStorage.removeItem('token')
    localStorage.removeItem('userInfo')
  }
}

const actions = {
  login({ commit }, loginForm) {
    return new Promise((resolve, reject) => {
      userApi.login(loginForm).then(res => {
        commit('SET_TOKEN', res.data.token)
        commit('SET_USER_INFO', res.data.user)
        resolve(res)
      }).catch(err => {
        reject(err)
      })
    })
  },
  logout({ commit }) {
    commit('CLEAR_USER')
  }
}
```

该模块的状态（State）从`localStorage`中初始化，确保页面刷新后用户的登录状态不会丢失。`login`动作（Action）调用后端登录接口，成功后将令牌和用户信息分别通过`SET_TOKEN`和`SET_USER_INFO`变更（Mutation）同步更新到Vuex状态和`localStorage`中，实现内存与本地存储的双重持久化。`logout`动作通过`CLEAR_USER`变更清除Vuex状态和`localStorage`中的所有用户数据。这种设计确保了用户登录状态的持久性和一致性，Vuex中的状态用于组件间的响应式共享，`localStorage`中的数据用于跨页面刷新的状态恢复。
