# 第4章 系统设计

本章将从系统总体架构、功能模块、数据库和接口四个维度，对萌宠乐园在线管理平台进行全面的设计阐述。系统设计以需求分析为基础，遵循软件工程的设计原则，力求在功能完备性、系统可扩展性和用户体验之间取得平衡。

## 4.1 系统总体架构设计

### 4.1.1 系统架构概述

萌宠乐园在线管理平台采用B/S（Browser/Server）三层架构模式进行设计与开发。B/S架构具有部署简便、维护成本低、跨平台访问等优势，用户仅需通过浏览器即可访问系统的全部功能，无需安装任何客户端软件。系统的三层架构分别如下：

（1）表现层（前端）

表现层负责与用户进行直接交互，承担数据展示与用户输入收集的职责。本系统前端采用Vue.js渐进式JavaScript框架，结合Element UI组件库构建用户界面。Vue.js采用组件化开发模式，通过声明式渲染和响应式数据绑定机制，实现了数据与视图的高效同步。Vue Router负责前端路由管理，实现了单页面应用（SPA）的无刷新页面切换。Vuex作为集中式状态管理工具，统一管理应用的全局状态。前端通过Axios库发起HTTP请求，以JSON格式与后端进行数据交互。

（2）业务逻辑层（后端）

业务逻辑层是系统的核心层，负责接收前端请求、执行业务逻辑处理并返回处理结果。本系统后端采用Spring Boot框架，其内嵌Tomcat服务器，提供了自动配置、起步依赖等特性，极大地简化了项目的搭建与部署过程。后端以RESTful风格对外提供API接口，统一采用JSON格式进行数据传输。在安全认证方面，系统采用JWT（JSON Web Token）机制实现无状态的用户身份认证，通过拦截器对需要认证的请求进行Token校验。MyBatis-Plus作为数据访问层的增强工具，在MyBatis的基础上提供了通用CRUD操作、分页插件、条件构造器等功能，显著减少了数据访问层的代码量。

（3）数据访问层

数据访问层负责与底层数据库进行交互，完成数据的持久化存储与查询操作。本系统采用MySQL关系型数据库作为数据存储引擎，使用utf8mb4字符集以支持包括Emoji表情在内的完整Unicode字符。数据库表采用InnoDB存储引擎，支持事务处理、行级锁定和外键约束，保证了数据的完整性与一致性。MyBatis-Plus通过配置下划线到驼峰命名的自动映射，实现了数据库字段与Java对象属性的无缝转换。

前后端之间通过HTTP协议进行通信，数据交换格式统一采用JSON。前端发送HTTP请求至后端API接口，后端经过业务处理后返回统一的响应格式（包含状态码、消息和数据），实现了前后端的完全解耦。

### 4.1.2 系统架构图

[图4-1 系统总体架构图]

如图4-1所示，系统总体架构自上而下分为表现层、业务逻辑层和数据访问层三个层次。表现层运行在用户浏览器端，通过Vue.js框架渲染页面，利用Axios发起HTTP请求。业务逻辑层运行在服务器端的Spring Boot应用中，包含控制器（Controller）、服务（Service）和数据访问对象（Mapper）三个子层。控制器负责接收和解析HTTP请求，服务层封装具体的业务逻辑，数据访问对象通过MyBatis-Plus与数据库进行交互。数据访问层由MySQL数据库组成，负责数据的持久化存储。

三层之间通过接口进行通信，各层职责明确、耦合度低。表现层仅关注数据的展示与交互，不涉及任何业务逻辑；业务逻辑层封装了全部的业务处理规则，对上层提供统一的API接口；数据访问层专注于数据的读写操作，对上层屏蔽了底层数据库的具体实现细节。这种分层架构使得各层可以独立进行开发、测试和维护，当某一层需要修改或替换时，不会对其他层产生较大影响。

## 4.2 系统功能模块设计

### 4.2.1 功能模块总体设计

[图4-3 系统功能模块图]

根据需求分析的结果，系统功能模块按照用户角色的不同，划分为前台用户子系统和后台管理子系统两大部分，如图4-3所示。

前台用户子系统面向普通注册用户，提供以下六个核心功能模块：

（1）首页展示模块：展示系统公告、热门宠物推荐、服务项目介绍、热销商品展示等内容，为用户提供一站式的信息浏览入口。

（2）宠物浏览模块：提供宠物信息的分类浏览和关键词搜索功能，用户可以查看宠物的详细信息，包括品种、年龄、性别、健康状态等。

（3）服务预约模块：用户可以浏览系统提供的服务类型（如洗澡美容、宠物寄养、宠物医疗、宠物训练等），选择宠物并填写预约信息，提交服务预约申请。

（4）商城购物模块：用户可以浏览和搜索商品，查看商品详情，管理购物车，提交订单并完成购买流程。

（5）社区论坛模块：用户可以发布帖子分享养宠经验，浏览其他用户的帖子，通过评论进行互动交流。

（6）个人中心模块：用户可以管理个人信息（头像、昵称、联系方式等），查看自己的预约记录、订单记录和发帖记录。

后台管理子系统面向系统管理员，提供以下七个核心功能模块：

（1）数据统计模块：展示系统运营的核心数据指标，包括用户总数、宠物数量、订单数量、预约数量、帖子数量及今日新增订单等。

（2）用户管理模块：管理员可以查看用户列表、搜索用户、修改用户信息、禁用或启用用户账号。

（3）宠物管理模块：管理员可以对宠物信息进行增删改查操作，管理宠物分类，控制宠物的上架与下架状态。

（4）预约管理模块：管理员可以查看所有预约记录，对预约进行确认、完成、拒绝等状态流转操作。

（5）商品管理模块：管理员可以对商品信息进行增删改查操作，管理商品分类，管理商品库存。

（6）订单管理模块：管理员可以查看所有订单信息，对订单进行支付确认、发货、完成等状态管理。

（7）社区管理模块：管理员可以对帖子进行置顶、隐藏、删除等操作，管理评论内容。

### 4.2.2 用户管理模块设计

用户管理模块是系统安全的基础保障，涵盖用户注册、用户登录和JWT认证三个核心流程。

用户注册流程设计如下：用户在前端页面填写注册信息（用户名、密码、昵称、邮箱、手机号等），前端对输入信息进行格式校验后，将数据以JSON格式提交至后端注册接口。后端接收到注册请求后，首先通过用户名在数据库中查询该用户名是否已被注册，若已存在则返回错误提示；若不存在，则使用Spring Security的BCrypt算法对用户密码进行加密处理，将加密后的密码与用户信息一同保存至数据库。密码加密存储有效防止了用户隐私数据的泄露风险。

用户登录流程设计如下：用户输入用户名和密码，后端接收登录请求后，根据用户名查询数据库获取用户记录，然后使用BCrypt算法对输入密码与数据库中存储的加密密码进行比对验证。若验证通过，则使用JWT库生成一个包含用户ID和角色信息的Token令牌，设置有效期为24小时，将Token与用户基本信息（用户ID、用户名、昵称、头像、角色等）一并返回给前端。前端接收到登录成功响应后，将Token存储在localStorage中，后续请求在HTTP请求头的Authorization字段中携带该Token。

JWT认证流程设计如下：后端配置了JWT拦截器，对所有需要认证的API请求进行拦截。拦截器从请求头的Authorization字段中提取Token，使用密钥对Token进行解析和验证。若Token有效，则将解析出的用户ID和角色信息存入HttpServletRequest的属性中，请求放行至后续的控制器方法；若Token无效或已过期，则返回认证失败的响应，前端收到认证失败响应后跳转至登录页面。

### 4.2.3 宠物信息管理模块设计

宠物信息管理模块是系统的核心业务模块之一，负责宠物信息的全生命周期管理。模块设计包含以下三个方面的内容：

（1）宠物CRUD操作设计

宠物信息的增删改查操作遵循RESTful设计规范。管理员可以通过后台管理界面新增宠物信息，填写宠物名称、所属分类、品种、年龄、性别、照片、描述和健康状态等字段，系统将信息保存至t_pet表。管理员可以对已有宠物信息进行编辑修改或删除操作。普通用户可以在前台浏览宠物列表和查看宠物详情。宠物列表支持分页展示，每页默认显示10条记录。

（2）分类管理设计

系统对宠物分类和商品分类采用独立管理的策略，分别通过t_pet_category表和t_product_category表进行存储。宠物分类管理提供分类的增删改查操作，每个分类包含分类名称、排序值和状态三个核心字段。排序值用于控制分类在前端页面的展示顺序，状态字段用于控制分类的启用与禁用。当前系统预设了猫类、狗类、小宠、水族、鸟类五个宠物分类。

（3）分页查询设计

宠物列表的分页查询支持按分类ID进行筛选和按关键词进行搜索。后端接收前端传递的页码、每页条数、分类ID和关键词参数后，使用MyBatis-Plus的条件构造器动态拼接查询条件。当分类ID参数不为空时，添加分类ID的等值查询条件；当关键词参数不为空时，添加宠物名称的模糊查询条件。最终通过MyBatis-Plus的分页插件执行分页查询，返回包含总记录数、总页数和当前页数据的分页结果。

### 4.2.4 服务预约模块设计

服务预约模块为用户提供便捷的宠物服务预约功能，其业务流程和状态管理设计如下：

预约流程设计：用户首先在前台浏览系统提供的服务类型列表，了解各项服务的名称、描述和价格信息。选择目标服务后，用户选择需要服务的宠物，填写预约日期和备注信息，提交预约申请。系统将预约信息保存至t_appointment表，初始状态设置为"待确认"。管理员在后台查看新提交的预约申请，根据实际情况对预约进行确认或拒绝操作。确认后的预约在服务完成后，由管理员将状态更新为"已完成"。用户也可以在预约确认前主动取消预约。

预约状态流转设计：预约记录共有五种状态，其流转规则如下：

- 待确认（0）：用户提交预约后的初始状态
- 已确认（1）：管理员审核通过，确认预约有效
- 已完成（2）：服务已执行完毕
- 已取消（3）：用户主动取消预约
- 已拒绝（4）：管理员审核后拒绝预约

状态之间的合法流转路径为：待确认→已确认→已完成（正常流程），待确认→已取消（用户取消），待确认→已拒绝（管理员拒绝）。已取消、已拒绝和已完成为终态，不可再进行状态变更。

### 4.2.5 商品与订单模块设计

商品与订单模块实现了从商品浏览到完成购买的完整电商交易流程。

购物流程设计：用户可以在前台商城页面浏览商品列表，支持按商品分类筛选和按关键词搜索。用户查看商品详情后，可以将商品加入购物车。在购物车页面，用户可以调整商品数量、删除商品或选择结算。结算时用户填写收货地址、联系电话和备注信息，确认无误后提交订单。系统根据购物车中的商品信息生成订单主记录和订单明细记录，同时扣减对应商品的库存数量。

订单状态流转设计：订单共有五种状态，其流转规则如下：

- 待支付（0）：订单创建后的初始状态
- 已支付（1）：用户完成支付（或管理员确认支付）
- 已发货（2）：管理员确认发货
- 已完成（3）：用户确认收货
- 已取消（4）：订单被取消

状态之间的合法流转路径为：待支付→已支付→已发货→已完成（正常流程），待支付→已取消（取消订单）。已取消和已完成为终态。

订单明细设计：系统采用主从表结构管理订单数据。订单主表（t_order）记录订单的核心信息，包括订单号、用户ID、总金额、订单状态、收货地址、联系电话和备注。订单明细表（t_order_item）记录订单中每个商品的详细信息，包括商品ID、商品名称、商品图片、购买单价和购买数量。一个订单主记录可以对应多条订单明细记录，实现了一对多的关联关系。在订单创建时，系统将购物车中每个商品的信息快照至订单明细表中，即使后续商品信息发生变化，历史订单中的商品信息仍然保持创建时的状态。

### 4.2.6 社区论坛模块设计

社区论坛模块为用户提供一个分享养宠经验、交流互动的社区平台，其功能设计如下：

帖子功能设计：登录用户可以在论坛中发布帖子，填写标题、正文内容，并可上传多张图片（图片以逗号分隔的URL形式存储）。帖子列表支持分页浏览和关键词搜索，按发布时间倒序排列，置顶帖子优先显示。用户可以查看帖子详情，系统会自动记录帖子的浏览量。帖子的修改和删除操作设置了权限控制：普通用户只能操作自己发布的帖子，管理员可以操作所有帖子。

评论功能设计：用户可以在帖子详情页面发表评论，评论内容与帖子通过帖子ID进行关联。评论列表按创建时间正序排列，展示评论者的昵称、头像、评论内容和评论时间。管理员拥有删除任何评论的权限，以维护社区的健康发展。

管理员审核功能设计：管理员可以对帖子进行置顶操作，将重要的帖子固定在列表顶部，提高其曝光度。管理员也可以对违规帖子进行隐藏处理，隐藏后的帖子不会在前台列表中展示。此外，管理员可以直接删除违规帖子和评论。

## 4.3 数据库设计

### 4.3.1 数据库设计原则

数据库设计是系统设计的核心环节，直接关系到系统的数据存储效率、查询性能和数据一致性。本系统的数据库设计遵循以下原则：

（1）规范化原则：数据库设计严格遵循第三范式（3NF），消除数据冗余，确保每个非主属性都完全依赖于主键，不存在传递依赖。在保证规范化设计的同时，对部分高频查询场景进行适度的反范式设计，以提高查询性能。

（2）命名规范原则：所有数据表名采用t_前缀加下划线命名法（如t_user、t_pet_category），字段名统一使用下划线命名法（如create_time、category_id）。这种命名规范与MyBatis-Plus的下划线到驼峰自动映射功能相配合，实现了数据库字段与Java对象属性的无缝对应。

（3）主键设计原则：所有数据表的主键采用BIGINT类型自增策略，BIGINT类型提供了足够大的取值范围，能够满足系统长期运营的数据增长需求。自增主键有利于B+树索引的顺序插入，提高了写入性能。

（4）时间戳原则：所有业务表统一包含create_time（创建时间）和update_time（更新时间）字段。create_time字段默认值为CURRENT_TIMESTAMP，在记录插入时自动填充；update_time字段默认值为CURRENT_TIMESTAMP，并在记录更新时自动刷新为当前时间。时间戳字段为数据审计、问题排查和统计分析提供了基础数据支撑。

（5）字符集与引擎原则：数据库统一使用utf8mb4字符集，该字符集是UTF-8的完整实现，支持包括Emoji表情在内的四字节Unicode字符，能够满足社区论坛等场景中用户输入多样化内容的需求。所有数据表使用InnoDB存储引擎，该引擎支持事务处理（ACID特性）、行级锁定和外键约束，适用于高并发读写场景。

### 4.3.2 E-R图设计

如图4-7所示，系统的实体关系（E-R）模型包含以下主要实体及其关联关系：

（1）用户与预约：一对多关系。一个用户可以创建多个服务预约记录，每个预约记录归属于一个用户。关联字段为t_appointment表中的user_id。

（2）用户与订单：一对多关系。一个用户可以创建多个订单，每个订单归属于一个用户。关联字段为t_order表中的user_id。

（3）用户与帖子：一对多关系。一个用户可以发布多个帖子，每个帖子归属于一个用户。关联字段为t_post表中的user_id。

（4）用户与评论：一对多关系。一个用户可以发表多条评论，每条评论归属于一个用户。关联字段为t_comment表中的user_id。

（5）宠物分类与宠物：一对多关系。一个宠物分类下包含多个宠物信息，每个宠物归属于一个宠物分类。关联字段为t_pet表中的category_id。

（6）商品分类与商品：一对多关系。一个商品分类下包含多个商品，每个商品归属于一个商品分类。关联字段为t_product表中的category_id。

（7）订单与订单明细：一对多关系。一个订单包含多个订单明细项，每个订单明细项归属于一个订单。关联字段为t_order_item表中的order_id。

（8）商品与订单明细：一对多关系。一个商品可以出现在多个订单明细中，每个订单明细项对应一个商品。关联字段为t_order_item表中的product_id。

（9）帖子与评论：一对多关系。一个帖子可以有多条评论，每条评论归属于一个帖子。关联字段为t_comment表中的post_id。

（10）服务类型与预约：一对多关系。一种服务类型可以被多次预约，每个预约记录对应一种服务类型。关联字段为t_appointment表中的service_id。

（11）宠物与预约：一对多关系。一只宠物可以有多个预约记录，每个预约记录关联一只宠物。关联字段为t_appointment表中的pet_id。

## 4.4 接口设计

### 4.4.1 接口设计规范

本系统后端接口采用RESTful架构风格进行设计，遵循以下规范：

（1）统一前缀：所有API接口的访问路径统一以/api作为前缀，通过Spring Boot的context-path配置实现。后端服务运行在8088端口，完整的接口基础路径为http://域名:8088/api/。

（2）HTTP方法语义：使用标准的HTTP方法表达操作语义。GET方法用于查询资源，POST方法用于创建资源，PUT方法用于更新资源，DELETE方法用于删除资源。

（3）统一响应格式：所有接口返回统一的JSON响应格式，通过Result<T>泛型类封装。响应包含三个字段：code（状态码，200表示成功，500表示业务异常，403表示权限不足）、message（响应消息）和data（响应数据，使用泛型支持任意类型）。成功响应的code为200，message为"success"；失败响应的code为500或自定义错误码，message为具体的错误描述信息。

（4）JSON数据交互：请求体和响应体统一使用JSON格式进行数据序列化与反序列化。Spring Boot默认使用Jackson作为JSON处理框架，日期格式统一配置为yyyy-MM-dd HH:mm:ss，时区设置为GMT+8。

（5）JWT Token认证：需要用户认证的接口，前端在请求头的Authorization字段中以"Bearer {token}"的格式携带JWT Token。后端通过拦截器对Token进行解析和验证，将用户ID和角色信息注入到请求上下文中，供后续的业务逻辑使用。

## 4.5 本章小结

本章从系统总体架构、功能模块、数据库和接口四个层面对萌宠乐园在线管理平台进行了详细的系统设计。系统采用B/S三层架构，前后端分离开发，通过RESTful API和JSON格式进行数据通信。功能模块涵盖前台用户子系统和后台管理子系统，共包含13个功能模块。数据库共设计了11张数据表，遵循第三范式设计原则。接口设计采用RESTful风格，通过统一的响应格式和JWT认证机制保障了系统的规范性和安全性。本章的设计成果为后续的系统实现提供了明确的技术蓝图和实施依据。
