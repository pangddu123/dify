# 第3章 系统需求分析

系统需求分析是软件开发过程中至关重要的阶段，其目的在于明确系统需要实现的功能和达到的性能指标，为后续的系统设计与实现奠定基础。本章将从系统可行性分析、用户角色分析、功能需求分析和非功能需求分析四个维度，对萌宠乐园在线管理平台进行全面的需求分析。

## 3.1 系统可行性分析

在系统开发启动之前，有必要从技术、经济和操作三个层面对项目进行可行性评估，以确保项目能够在既定的资源和时间约束下顺利实施。

### 3.1.1 技术可行性

本系统后端采用Spring Boot框架作为核心开发框架，该框架基于成熟的Spring生态体系，具备自动配置、开箱即用等特点，能够显著降低项目搭建和配置的复杂度。数据持久层采用MyBatis-Plus作为ORM框架，MyBatis-Plus在MyBatis的基础上进行了增强，提供了通用的CRUD操作、条件构造器和分页插件等功能，能够有效减少重复的SQL编写工作，提高开发效率。数据库选用MySQL关系型数据库，该数据库是全球应用最为广泛的开源数据库之一，具备高性能、高可靠性和良好的扩展性，完全能够满足本系统的数据存储需求。

前端采用Vue.js渐进式JavaScript框架，配合Element UI组件库进行界面开发。Vue.js采用组件化开发模式，具备响应式数据绑定和虚拟DOM等技术优势，能够构建交互丰富、性能优良的单页面应用。Element UI是由饿了么前端团队开源的一套基于Vue.js的桌面端组件库，提供了丰富的表单、表格、弹窗等UI组件，能够确保界面风格的一致性和交互体验的友好性。

在系统安全方面，采用JWT（JSON Web Token）实现无状态的用户认证机制。JWT令牌在用户登录成功后由服务端签发，客户端在后续请求中携带该令牌进行身份验证，避免了传统Session机制在分布式环境下的状态同步问题，同时降低了服务端的存储开销。

上述技术栈均为业界广泛使用的开源技术，拥有活跃的开发者社区和完善的官方文档，技术成熟度高，风险可控。综合以上分析，本系统在技术层面具有良好的可行性。

### 3.1.2 经济可行性

本系统所采用的全部技术栈均为开源免费的软件产品。Spring Boot、MyBatis-Plus、Vue.js、Element UI等框架和组件库均遵循开源协议，无需支付任何许可费用。MySQL数据库的社区版同样可以免费使用，能够满足中小规模应用的数据存储需求。

在开发工具方面，后端开发可使用IntelliJ IDEA社区版（免费）或Eclipse等免费IDE，前端开发可使用Visual Studio Code等免费编辑器，版本管理可使用Git等免费工具。因此，系统开发阶段无需投入额外的软件采购成本。

在部署运维方面，本系统基于B/S架构，可部署在普通的云服务器上运行，如阿里云ECS、腾讯云CVM等基础实例即可满足运行需求，部署成本较低。此外，Spring Boot应用内置了Tomcat容器，支持以JAR包方式直接运行，进一步简化了部署流程，降低了运维成本。

综合以上分析，本系统的开发和运维成本可控，在经济层面具有良好的可行性。

### 3.1.3 操作可行性

本系统采用B/S（Browser/Server）架构，用户无需安装任何客户端软件，仅需通过Web浏览器即可访问系统的全部功能。这种架构模式消除了客户端部署和更新的繁琐工作，降低了用户的使用门槛。

在界面设计方面，前端采用Element UI组件库提供了一套风格统一、交互规范的UI组件，包括表单输入、数据表格、分页导航、消息提示等常用组件，能够为用户提供一致且直观的操作体验。系统界面布局清晰、功能模块划分合理，用户能够快速定位所需功能并完成相应操作。

此外，系统支持响应式页面设计，能够在不同尺寸的屏幕上呈现合理的布局效果，适配桌面电脑、平板电脑等多种终端设备，进一步提升了系统的可访问性和易用性。

综合以上分析，本系统在操作层面具有良好的可行性，用户能够以较低的学习成本快速上手使用。

## 3.2 用户角色分析

通过对萌宠乐园在线管理平台的业务场景进行分析，本系统涉及两类核心用户角色：普通用户和管理员。两类角色在系统中承担不同的职责，拥有不同的功能权限。

**普通用户**是平台的主要服务对象，其核心需求包括：注册并登录系统以获取个人账户；浏览平台展示的各类宠物信息，了解宠物的品种、年龄、健康状况等详细资料；根据自身需求预约宠物服务，包括洗澡美容、寄养、医疗、训练等服务类型；浏览平台商城中的宠物用品和饲料等商品，将所需商品加入购物车并完成在线下单购买；在社区论坛中发布养宠经验分享帖、浏览其他用户发布的帖子并进行评论互动；管理个人基本信息、查看历史订单记录和预约记录。

**管理员**是平台的运营管理人员，负责平台的日常管理和维护工作。管理员除了具备普通用户的基本功能外，还拥有后台管理的专属权限，其核心职责包括：管理平台注册用户，包括查看用户列表、编辑用户信息以及禁用或启用用户账户；管理宠物信息，包括添加新宠物、编辑宠物资料、删除宠物信息以及管理宠物分类；处理用户提交的服务预约，对预约进行确认、拒绝或标记完成等操作；管理平台商品，包括商品的添加、编辑、删除和上下架管理，以及商品分类的维护；管理用户订单，跟踪订单状态并进行发货、完成等操作；管理社区论坛内容，包括帖子的审核、置顶、隐藏和删除，以及评论的管理；通过数据统计看板查看平台的运营数据，包括用户总数、宠物数量、订单数量、预约数量等关键指标。

两类用户角色的功能权限划分体现了系统设计中"权责分离"的原则，既保证了普通用户的便捷使用体验，又确保了管理员对平台的有效管控。

## 3.3 功能需求分析

基于对用户角色和业务场景的深入分析，本系统的功能需求可划分为六个核心模块：用户管理模块、宠物信息管理模块、服务预约模块、商品与订单模块、社区论坛模块和后台管理模块。下面逐一进行详细的功能需求分析。

### 3.3.1 用户管理模块

用户管理模块是系统的基础模块，承担用户身份认证和个人信息管理的核心功能。该模块面向普通用户和管理员两类角色，提供差异化的功能支持。

对于普通用户，该模块提供注册、登录、查看个人信息和修改个人信息四项主要功能。用户注册时需提供用户名和密码，系统对用户名的唯一性进行校验，并对密码进行加密存储以保障账户安全。用户登录时，系统验证用户名和密码的正确性，校验通过后签发JWT令牌，客户端在后续请求中携带该令牌进行身份识别。登录成功后，用户可以查看和修改个人资料，包括昵称、邮箱、手机号和头像等信息。

对于管理员，该模块提供用户列表查看、用户信息编辑和用户账户禁用/启用等管理功能。管理员可以按照关键词搜索用户，查看用户的详细信息和账户状态，并可根据运营需要对违规用户进行账户禁用处理。

### 3.3.2 宠物信息管理模块

宠物信息管理模块是平台的核心业务模块之一，负责宠物信息的展示和管理。该模块包括宠物信息浏览、宠物详情查看、宠物分类管理等功能。

对于普通用户，该模块提供宠物列表浏览和宠物详情查看功能。宠物列表支持按分类进行筛选浏览，同时支持按关键词搜索宠物名称或品种。用户可以点击宠物卡片进入宠物详情页面，查看宠物的名称、品种、年龄、性别、照片、健康状态和详细描述等完整信息。

对于管理员，该模块提供宠物的完整生命周期管理功能，包括添加新宠物、编辑宠物信息、删除宠物以及上下架管理。管理员可以创建和管理宠物分类，对分类进行排序和启用/禁用操作。每个宠物需关联一个分类，包含名称、品种、年龄、性别、照片、描述和健康状态等属性信息。宠物设有上架和下架两种状态，下架的宠物不会在前台展示。

### 3.3.3 服务预约模块

服务预约模块是连接用户与宠物服务的重要功能模块。该模块允许用户选择特定的宠物和服务类型，提交服务预约申请，并由管理员对预约进行处理。

系统预设了多种宠物服务类型，包括洗澡美容、寄养、医疗、训练等，每种服务类型包含名称、描述和价格信息。用户在浏览宠物详情时，可以选择该宠物并预约相应的服务，填写预约日期和备注信息后提交预约申请。预约提交后，初始状态为"待确认"。

用户可以在"我的预约"页面查看自己提交的所有预约记录及其当前状态。对于尚未确认的预约，用户可以主动取消。

管理员可以在后台查看所有用户提交的预约列表，支持按状态进行筛选。管理员可以对预约进行确认、拒绝、标记完成等操作。预约状态流转为：待确认→已确认→已完成，用户也可将预约取消（状态变为已取消），管理员可拒绝预约（状态变为已拒绝）。

### 3.3.4 商品与订单模块

商品与订单模块实现了平台的电商功能，包括商品的展示浏览、购物车管理和订单处理等完整交易流程。

对于普通用户，该模块提供商品浏览、商品详情查看、下单购买和订单管理功能。商品列表支持按商品分类筛选和关键词搜索，每个商品展示名称、价格、图片和销量等关键信息。用户点击商品可查看详情，包括商品描述、库存和完整的产品图片。用户下单时，选择需要购买的商品及其数量，填写收货地址和联系电话，提交后系统自动生成唯一订单号，计算订单总金额，并创建订单记录和订单明细记录。订单初始状态为"待支付"。用户可以在"我的订单"页面查看所有历史订单及其状态。

对于管理员，该模块提供商品管理和订单管理两项核心功能。在商品管理方面，管理员可以进行商品的添加、编辑、删除操作，管理商品分类，以及控制商品的上架和下架状态。在订单管理方面，管理员可以查看所有用户订单的列表，查看订单详情，并对订单进行状态流转操作，包括确认支付（状态变为"已支付"）、发货（状态变为"已发货"）和标记完成（状态变为"已完成"）。订单状态完整流转为：待支付→已支付→已发货→已完成，用户也可以取消订单。

### 3.3.5 社区论坛模块

## 3.4 非功能需求分析

除功能需求外，系统还需要满足一系列非功能性需求，以确保系统在性能、安全性、可用性和可维护性等方面达到预期的质量标准。

### 3.4.1 性能需求

系统的性能直接影响用户体验，本系统在性能方面提出以下需求：

（1）页面响应时间：系统页面的平均加载和响应时间应控制在3秒以内，确保用户操作的流畅性。对于数据量较大的列表查询，通过分页加载机制控制单次返回的数据量，避免一次性加载过多数据导致的性能瓶颈。

（2）并发处理能力：系统应支持至少100个并发用户同时在线访问，在并发场景下系统的响应时间不应出现显著退化。Spring Boot内置的Tomcat容器支持多线程请求处理，配合数据库连接池技术（如HikariCP），能够有效应对一定规模的并发访问。

（3）数据库查询优化：针对频繁查询的数据表，通过建立合理的索引提高查询效率。例如，宠物表和商品表均建立了分类ID索引（idx_category_id），预约表建立了用户ID、宠物ID和服务ID索引，订单表建立了用户ID索引和订单号唯一索引。这些索引设计能够显著提升按条件查询和关联查询的性能。

### 3.4.2 安全性需求

系统安全是保障用户数据和平台运营安全的基础，本系统在安全性方面提出以下需求：

（1）用户密码加密存储：用户密码不以明文形式存储在数据库中，而是采用MD5加盐值（Salt）的方式进行加密处理。即使数据库遭到泄露，攻击者也无法直接获取用户的原始密码，有效保护用户的账户安全。

（2）JWT令牌认证：系统采用JWT（JSON Web Token）实现无状态的用户认证机制。用户登录成功后，服务端签发包含用户ID和角色信息的JWT令牌，客户端在后续请求的Header中携带该令牌。服务端通过拦截器对请求进行统一的令牌校验，确保只有持有有效令牌的用户才能访问受保护的接口资源，有效防止未授权访问。

（3）管理员接口权限校验：系统对管理员专属接口进行严格的权限校验。在帖子管理的置顶、隐藏等操作中，后端控制器会从请求属性中获取用户角色信息，判断当前用户是否为管理员（role=1），非管理员用户执行此类操作将被拒绝并返回403错误码，确保管理功能的访问安全。

（4）CORS跨域安全配置：前后端分离架构下，前端应用和后端API通常部署在不同的域名或端口上。系统通过配置CORS（Cross-Origin Resource Sharing）跨域策略，明确指定允许跨域访问的源地址、请求方法和请求头，既保证了前后端的正常通信，又防止了非法的跨域请求。

（5）SQL注入防护：系统采用MyBatis-Plus框架进行数据持久化操作，该框架默认使用参数化查询（PreparedStatement）方式执行SQL语句，预编译的SQL语句将用户输入作为参数传递，而非直接拼接在SQL字符串中，从根本上杜绝了SQL注入攻击的风险。

### 3.4.3 可用性需求

系统的可用性关乎用户的实际使用体验，本系统在可用性方面提出以下需求：

（1）界面友好性：系统界面采用Element UI组件库进行统一设计，保持视觉风格和交互方式的一致性。页面布局采用通用的顶部导航加侧边栏布局模式，功能模块划分清晰，用户能够快速定位目标功能。关键操作按钮和入口在页面中的位置醒目，减少用户的查找成本。

（2）错误提示明确性：当用户操作出现异常时，系统能够提供明确的错误提示信息，告知用户具体的错误原因和解决建议。例如，表单提交时必填字段未填写，系统会高亮显示未填写的字段并提示具体的字段名称；登录失败时，系统提示"用户名或密码错误"而非模糊的系统错误信息。

（3）表单输入验证：系统对用户的输入数据进行前端和后端双重验证。前端通过Element UI的表单验证规则，在用户输入时即进行格式校验，如邮箱格式、手机号格式、必填项检查等，减少无效请求的提交。后端对关键参数进行二次校验，确保数据的完整性和合法性，防止绕过前端验证的恶意请求。

### 3.4.4 可维护性需求

系统的可维护性决定了系统后续迭代和运维的效率，本系统在可维护性方面提出以下需求：

（1）分层架构设计：系统后端采用经典的三层架构模式，分为Controller层、Service层和Mapper层。Controller层负责接收和响应HTTP请求，Service层负责业务逻辑处理，Mapper层负责与数据库进行交互。各层之间职责清晰、耦合度低，便于独立开发和维护。当某一层需要修改时，只要接口契约不变，其他层无需调整。

（2）统一异常处理：系统采用全局异常处理机制，通过Spring的@ControllerAdvice注解实现统一的异常捕获和响应格式化。业务异常和系统异常被统一封装为标准的错误响应格式返回给前端，避免异常信息直接暴露给用户，同时简化了各业务模块中的异常处理代码。

（3）统一响应格式：系统定义了统一的响应封装类Result，所有接口返回值均采用该类进行封装，包含状态码、提示信息和数据内容三个标准字段。这种统一的响应格式使前端能够以一致的方式解析和处理接口响应，降低了前后端对接的复杂度。

（4）配置文件外置：系统采用Spring Boot的外部化配置机制，数据库连接信息、服务端口、文件上传路径等可配置项均定义在application.yml配置文件中。在不同环境（开发、测试、生产）下部署时，仅需修改配置文件而无需改动代码，提高了系统的部署灵活性和环境适配能力。

## 3.5 本章小结

本章对萌宠乐园在线管理平台进行了全面的系统需求分析。首先，从技术可行性、经济可行性和操作可行性三个维度论证了项目实施的可行性，确认项目具备良好的实施条件。其次，明确了系统涉及普通用户和管理员两类用户角色，并详细分析了各自的职责和功能权限。然后，将系统的功能需求划分为用户管理、宠物信息管理、服务预约、商品与订单、社区论坛和后台管理六个核心模块，逐一进行了详细的功能分析，并通过用例描述表格对各模块的核心用例进行了规范化描述。最后，从性能、安全性、可用性和可维护性四个方面提出了系统的非功能需求，为后续的系统设计与实现提供了明确的质量标准。本章的需求分析成果将作为第4章系统设计的直接输入依据。
