# 第3章 系统需求分析

系统需求分析是软件开发过程中至关重要的阶段，其目的在于明确系统需要实现的功能和应达到的性能指标，为后续的系统设计与实现奠定基础。本章从系统可行性、用户角色、功能需求和非功能需求四个维度，对萌宠乐园在线管理平台进行全面的需求分析。

## 3.1 系统可行性分析

在项目启动之前，有必要从技术、经济和操作三个层面对项目进行可行性评估，以确保项目能够在既定的资源与时间约束下顺利实施。

### 3.1.1 技术可行性

本系统采用Spring Boot + MyBatis-Plus + MySQL的后端技术栈和Vue.js + Element UI的前端技术栈（具体特性详见第2章），并通过JWT实现无状态身份认证。上述技术均为业界广泛使用的成熟开源方案，拥有活跃的社区支持与完善的官方文档，学习成本与落地风险均可控。开发团队对Java Web开发和前端框架具备必要的技术储备，能够按计划完成各模块开发与联调。综上，本系统在技术层面具备良好的可行性。

### 3.1.2 经济可行性

本系统所采用的全部技术栈均为开源免费软件。Spring Boot、MyBatis-Plus、Vue.js、Element UI等框架和组件库遵循开源协议，无需支付许可费用；MySQL社区版同样可免费使用，足以满足中小规模应用的数据存储需求。

在开发工具方面，后端开发可使用IntelliJ IDEA社区版或Eclipse等免费IDE，前端可使用Visual Studio Code等免费编辑器，版本管理使用Git，开发阶段无需额外的软件采购支出。在部署运维方面，系统基于B/S架构，可运行于阿里云ECS、腾讯云CVM等普通云服务器；Spring Boot内置Tomcat容器，以JAR包方式直接运行，进一步简化了部署流程、降低了运维成本。综上，本系统在经济层面具备良好的可行性。

### 3.1.3 操作可行性

本系统采用B/S（Browser/Server）架构，用户无需安装任何客户端软件，仅需通过浏览器即可访问全部功能，降低了使用门槛。前端基于Element UI组件库，界面风格统一、交互规范，用户能够以较低的学习成本快速上手。系统支持响应式布局，可适配桌面电脑和平板电脑等多种终端，进一步提升了可访问性和易用性。综上，本系统在操作层面具备良好的可行性。

## 3.2 用户角色分析

通过对萌宠乐园业务场景的梳理，本系统涉及两类核心用户角色：普通用户和管理员。两类角色在系统中承担不同职责、拥有不同的功能权限。

**普通用户**是平台的主要服务对象，核心需求包括：注册与登录账户；浏览平台展示的各类宠物信息，了解品种、年龄、健康状况等资料；根据自身需求预约宠物服务，包括洗澡美容、寄养、医疗、训练等；浏览商城中的宠物用品与饲料，加入购物车并完成在线下单；在社区论坛中发布养宠经验、浏览其他用户帖子并进行评论互动；管理个人基本信息、查看历史订单与预约记录。

**管理员**是平台的运营管理人员，除具备普通用户基本功能外，还拥有后台管理的专属权限：管理平台注册用户（查看、编辑、禁用/启用）；管理宠物信息（添加、编辑、删除以及分类维护）；处理用户提交的服务预约，进行确认、拒绝或标记完成等操作；管理平台商品及商品分类，进行上下架控制；管理用户订单，跟踪订单状态并执行发货、完成等操作；管理社区论坛内容，审核、置顶、隐藏或删除帖子及评论；通过数据统计看板查看用户数、宠物数、订单数、预约数等关键运营指标。

两类角色的权限划分体现了系统设计中"权责分离"的原则，既保证了普通用户的便捷使用体验，又确保了管理员对平台的有效管控。

## 3.3 功能需求分析

基于用户角色与业务场景分析，本系统的功能需求划分为六个核心模块：用户管理、宠物信息管理、服务预约、商品与订单、社区论坛和后台管理。下面逐一展开详细分析。

### 3.3.1 用户管理模块

用户管理模块是系统的基础模块，承担用户身份认证和个人信息管理的核心功能，面向普通用户和管理员两类角色提供差异化的能力。

对于普通用户，模块提供注册、登录、查看个人信息和修改个人信息四项主要功能。注册时需提供用户名和密码，系统校验用户名的唯一性并对密码进行加密存储；登录时验证用户名和密码，校验通过后签发JWT令牌供后续请求携带。登录成功后，用户可以查看和修改昵称、邮箱、手机号和头像等资料。

对于管理员，模块提供用户列表查看、用户信息编辑和用户账户禁用/启用等管理功能。管理员可按关键词搜索用户，查看详细信息与账户状态，并对违规用户进行禁用处理。

### 3.3.2 宠物信息管理模块

宠物信息管理模块负责宠物信息的展示与管理，包括宠物列表浏览、详情查看、分类管理等功能。

对于普通用户，模块提供列表浏览和详情查看功能。列表支持按分类筛选，并支持按关键词搜索名称或品种；用户可点击卡片进入详情页，查看名称、品种、年龄、性别、照片、健康状态与描述等完整信息。

对于管理员，模块提供宠物的全生命周期管理功能，包括新增、编辑、删除及上下架管理。管理员可以创建和维护宠物分类，支持排序与启用/禁用。每个宠物需关联一个分类，包含名称、品种、年龄、性别、照片、描述和健康状态等属性。下架的宠物不会在前台展示。

### 3.3.3 服务预约模块

服务预约模块是连接用户与宠物服务的重要环节，允许用户选择特定宠物和服务类型并提交预约申请，由管理员进行处理。

系统预设多种服务类型（包括洗澡美容、寄养、医疗、训练等），每种类型包含名称、描述和价格。用户在宠物详情页选择宠物和服务，填写预约日期和备注后提交，预约初始状态为"待确认"。用户可在"我的预约"页面查看预约记录与状态，对尚未确认的预约可主动取消。

管理员在后台查看所有预约，支持按状态筛选，可对预约进行确认、拒绝或标记完成等操作。预约状态流转为：待确认→已确认→已完成；用户可主动取消（已取消），管理员可拒绝（已拒绝）。

### 3.3.4 商品与订单模块

商品与订单模块实现了平台的电商功能，覆盖商品展示浏览、购物车管理和订单处理等完整交易流程。

对于普通用户，模块提供商品浏览、详情查看、下单购买和订单管理功能。商品列表支持分类筛选和关键词搜索，每个商品展示名称、价格、图片和销量等摘要信息。用户下单时选择商品及数量，填写收货地址和联系电话，系统自动生成唯一订单号，计算订单总金额，并创建订单及订单明细记录，初始状态为"待支付"。用户可在"我的订单"页面查看历史订单及其状态。

对于管理员，模块提供商品管理和订单管理两项核心功能。商品管理支持商品的增删改和分类维护，并可控制上下架；订单管理支持分页查看所有订单、查看详情，以及进行状态流转：确认支付（已支付）、发货（已发货）、标记完成（已完成）。订单状态完整流转为：待支付→已支付→已发货→已完成，用户也可以取消订单。

### 3.3.5 社区论坛模块

社区论坛模块为用户提供养宠经验交流与互动的平台，包括帖子发布、浏览以及评论互动等功能。

对于普通用户，模块提供帖子浏览、详情查看、发布帖子和评论帖子功能。帖子列表分页展示，支持按关键词搜索标题；详情页展示标题、正文、配图、作者、发布时间和浏览量。用户可发布新帖子（标题、正文、多张配图）并对帖子发表评论。

对于管理员，模块提供帖子和评论管理功能。管理员可以置顶重要帖子以优先展示，可以隐藏不当帖子使其不在前台显示，也可删除违规帖子或评论。编辑和删除操作均经过权限校验：普通用户只能操作自己发布的内容，管理员可操作全部内容。

### 3.3.6 后台管理模块

后台管理模块是管理员进行平台运营管理的统一入口，提供数据统计看板和系统管理功能。

管理员登录后台后可访问数据统计看板。看板以数字卡片形式直观展示平台核心运营数据，包括用户总数、宠物数量、订单总数、预约总数、帖子总数以及当日新增订单数等关键指标。这些统计数据由后端实时聚合查询获取，为管理员的运营决策提供数据支撑。

此外，后台管理模块还集成了上述各业务模块的管理入口（用户管理、宠物管理、预约管理、商品管理、订单管理、社区管理），形成统一的后台界面。

## 3.4 非功能需求分析

除功能需求外，系统还需在性能、安全性、可用性和可维护性等方面达到预期质量标准。

### 3.4.1 性能需求

（1）页面响应时间：系统页面的平均加载和响应时间应控制在3秒以内。对于数据量较大的列表查询，通过分页加载控制单次返回的数据量，避免一次性加载过多数据导致的性能瓶颈。

（2）并发处理能力：系统应支持至少100个并发用户同时在线访问，在并发场景下响应时间不应出现显著退化。Spring Boot内置的Tomcat容器支持多线程请求处理，配合数据库连接池（HikariCP）可应对一定规模的并发。

（3）数据库查询优化：针对高频查询字段建立合理的索引。例如，宠物表和商品表建立分类ID索引（idx_category_id），预约表建立用户ID、宠物ID和服务ID索引，订单表建立用户ID索引和订单号唯一索引，以显著提升按条件和关联查询的性能。

### 3.4.2 安全性需求

（1）用户密码加密存储：用户密码不以明文存储，采用MD5加盐值（Salt）的方式加密处理。即使数据库泄露，攻击者也无法直接获取原始密码，一定程度上保护了账户安全。

（2）JWT令牌认证：采用JWT实现无状态认证，登录成功后服务端签发包含用户ID和角色的令牌，客户端在后续请求的Header中携带。服务端通过拦截器统一校验令牌有效性，防止未授权访问。

（3）管理员接口权限校验：对管理员专属接口进行严格的权限校验。在帖子置顶、隐藏等敏感操作中，后端从请求属性中获取用户角色信息并判断是否为管理员（role=1），非管理员请求将被拒绝并返回403，以确保管理功能的访问安全。

（4）CORS跨域安全配置：系统通过配置CORS跨域策略明确指定允许跨域访问的源地址、请求方法和请求头，既保证了前后端的正常通信，又防止了非法的跨域请求。

（5）SQL注入防护：系统采用MyBatis-Plus进行数据持久化，默认使用参数化查询（PreparedStatement），将用户输入作为参数而非拼接到SQL字符串中，从根本上杜绝了SQL注入攻击风险。

### 3.4.3 可用性需求

（1）界面友好性：采用Element UI组件库统一设计，保持视觉风格与交互方式的一致性。页面采用"顶部导航+侧边栏"的通用布局，功能模块划分清晰，关键操作入口醒目，减少用户查找成本。

（2）错误提示明确性：当用户操作出现异常时，系统能够提供明确的错误提示信息，告知具体原因与解决建议。例如表单必填字段缺失时高亮显示并提示字段名称；登录失败时提示"用户名或密码错误"，避免模糊的系统错误信息。

（3）表单输入验证：系统对输入数据进行前后端双重验证。前端通过Element UI表单验证规则在输入时即进行格式校验（邮箱、手机号、必填项等），减少无效请求提交；后端对关键参数进行二次校验，防止绕过前端验证的恶意请求。

### 3.4.4 可维护性需求

（1）分层架构设计：系统后端采用经典三层架构，划分为Controller、Service和Mapper层，分别负责请求处理、业务逻辑和数据访问。各层职责清晰、耦合度低，便于独立开发和维护；在接口契约不变的前提下，某一层修改不会影响其他层。

（2）统一异常处理：通过Spring的@ControllerAdvice注解实现全局异常捕获与响应格式化，业务异常与系统异常统一封装为标准错误响应返回前端，避免异常信息直接暴露给用户，同时简化各业务模块的异常处理代码。

（3）统一响应格式：系统定义统一的响应封装类Result，所有接口返回值均包含状态码、提示信息和数据内容三个标准字段。统一的响应格式使前端能够以一致的方式解析和处理响应，降低了前后端对接的复杂度。

（4）配置文件外置：基于Spring Boot的外部化配置机制，数据库连接、服务端口、文件上传路径等可配置项均定义在application.yml中。切换开发、测试、生产环境时只需修改配置文件，无需改动代码，提高了部署灵活性。

## 3.5 本章小结

本章对萌宠乐园在线管理平台进行了全面的需求分析。首先从技术、经济、操作三个维度论证了项目的可行性；其次明确了普通用户与管理员两类角色的职责与权限；随后将系统功能划分为六个核心模块，对各模块进行了功能分析并辅以用例描述；最后从性能、安全性、可用性和可维护性四方面提出了非功能需求。本章的分析成果将作为第4章系统设计的直接输入依据。
