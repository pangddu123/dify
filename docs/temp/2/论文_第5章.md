# 第5章 系统实现

本章展示各功能模块的关键实现代码和界面效果。代码均从实际项目中提取，结合核心业务逻辑进行分析说明。

## 5.1 用户认证模块实现

用户认证模块是整个系统的安全基础，涵盖用户注册、登录、JWT令牌管理以及统一响应封装等功能。本模块采用前后端分离下的无状态认证方案，通过JWT实现用户身份的持久化验证。

### 5.1.1 用户注册实现

用户注册功能由`UserServiceImpl`类中的`register`方法实现，核心代码如下：

```java
@Override
public void register(User user) {
    LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(User::getUsername, user.getUsername());
    if (userMapper.selectOne(wrapper) != null) {
        throw new BusinessException("用户名已存在");
    }
    user.setPassword(SecureUtil.md5(user.getPassword() + SALT));
    if (user.getRole() == null) {
        user.setRole(0);
    }
    if (user.getStatus() == null) {
        user.setStatus(1);
    }
    userMapper.insert(user);
}
```

注册方法的执行流程如下：首先，通过`LambdaQueryWrapper`按用户名查询数据库，若已存在同名用户，则抛出`BusinessException`业务异常提示"用户名已存在"；通过唯一性校验后，调用Hutool工具库的`SecureUtil.md5()`方法对"密码+盐值"进行摘要运算。其中盐值`SALT`定义为常量字符串`"pet_kingdom_salt"`，可在一定程度上抵御彩虹表攻击和撞库破解。加密完成后，系统为新用户设置默认角色（`role=0`普通用户）和默认状态（`status=1`正常），最后通过MyBatis-Plus的`insert`方法持久化。

### 5.1.2 用户登录实现

用户登录功能同样由`UserServiceImpl`中的`login`方法实现，负责验证身份凭证并签发JWT令牌。核心代码如下：

```java
@Override
public Map<String, Object> login(String username, String password) {
    LambdaQueryWrapper<User> wrapper = new LambdaQueryWrapper<>();
    wrapper.eq(User::getUsername, username);
    User user = userMapper.selectOne(wrapper);
    if (user == null) {
        throw new BusinessException("用户名或密码错误");
    }
    if (!SecureUtil.md5(password + SALT).equals(user.getPassword())) {
        throw new BusinessException("用户名或密码错误");
    }
    if (user.getStatus() == 0) {
        throw new BusinessException("账号已被禁用");
    }
    String token = jwtUtil.generateToken(user.getId(), user.getUsername(), user.getRole());
    user.setPassword(null);
    Map<String, Object> result = new HashMap<>();
    result.put("token", token);
    result.put("user", user);
    return result;
}
```

登录方法分为四个关键步骤：（1）根据用户名查询用户实体，不存在时返回模糊提示"用户名或密码错误"，避免泄露用户是否存在的信息；（2）对输入密码进行同样的MD5加盐运算并与库中密文比对，不一致同样返回模糊提示；（3）检查账号状态，`status=0`表示已被禁用，拒绝登录；（4）全部校验通过后调用`JwtUtil.generateToken()`生成令牌，将令牌与脱敏用户信息（密码置空）封装为`Map`返回前端。这种设计避免了敏感信息的网络传输风险。

### 5.1.3 JWT工具类实现

JWT工具类`JwtUtil`封装了令牌的生成与验证逻辑，是系统无状态认证机制的核心组件，基于`auth0-java-jwt`库实现，关键代码如下：

```java
@Component
public class JwtUtil {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.expiration}")
    private long expiration;

    public String generateToken(Long userId, String username, Integer role) {
        Date now = new Date();
        Date expireDate = new Date(now.getTime() + expiration);
        return JWT.create()
                .withClaim("userId", userId)
                .withClaim("username", username)
                .withClaim("role", role)
                .withIssuedAt(now)
                .withExpiresAt(expireDate)
                .sign(Algorithm.HMAC256(secret));
    }

    public boolean verify(String token) {
        try {
            JWTVerifier verifier = JWT.require(Algorithm.HMAC256(secret)).build();
            verifier.verify(token);
            return true;
        } catch (TokenExpiredException e) {
            return false;
        } catch (JWTDecodeException e) {
            return false;
        }
    }
}
```

`generateToken`方法接收用户ID、用户名和角色三个参数作为令牌载荷（Payload），通过`JWT.create()`构建器生成令牌。令牌包含三个自定义声明（Claim）：`userId`用于识别用户身份、`username`用于展示用户信息、`role`用于权限判断；同时设置签发时间与过期时间（由配置项`jwt.expiration`控制）。令牌最终采用HMAC-SHA256算法和密钥签名，确保完整性和不可篡改性。

`verify`方法使用相同的密钥和算法构建验证器：令牌有效且未过期时返回`true`，过期（`TokenExpiredException`）或格式错误（`JWTDecodeException`）则返回`false`。此外，该类还提供`getUserId`、`getUsername`、`getRole`方法，通过解码令牌获取相应的声明值，供拦截器使用。

### 5.1.4 JWT拦截器实现

JWT拦截器`JwtInterceptor`实现了Spring MVC的`HandlerInterceptor`接口，在请求进入控制器前统一校验令牌：

```java
@Override
public boolean preHandle(HttpServletRequest request, HttpServletResponse response, Object handler) {
    if ("OPTIONS".equalsIgnoreCase(request.getMethod())) {
        return true;
    }

    String token = request.getHeader("Authorization");
    if (StringUtils.isBlank(token)) {
        token = request.getParameter("token");
    }

    if (StringUtils.isBlank(token)) {
        throw new BusinessException(401, "未登录，请先登录");
    }

    if (!token.startsWith("Bearer ")) {
        throw new BusinessException(401, "Token格式错误");
    }

    token = token.substring(7);
    if (!jwtUtil.verify(token)) {
        throw new BusinessException(401, "Token已过期或无效，请重新登录");
    }

    request.setAttribute("userId", jwtUtil.getUserId(token));
    request.setAttribute("username", jwtUtil.getUsername(token));
    request.setAttribute("role", jwtUtil.getRole(token));
    return true;
}
```

拦截器的执行逻辑分为五个阶段：首先对浏览器预检请求（OPTIONS方法）直接放行，这是处理跨域的标准做法；其次从请求头`Authorization`字段获取令牌，若为空则尝试从请求参数中获取；若仍未取得，抛出401异常提示未登录；接着校验令牌格式是否以`Bearer `前缀开头（OAuth 2.0 Bearer Token规范），并截取实际令牌字符串；最后调用`JwtUtil.verify()`验证令牌的有效性，通过后将用户ID、用户名和角色信息写入请求属性，供后续控制器通过`request.getAttribute()`获取。该设计使控制器无需关心认证细节，只需直接从请求属性中取用户上下文即可。

### 5.1.5 统一响应封装

系统采用统一的响应格式封装所有API返回结果，确保前端以一致方式处理响应数据。该封装由`Result`泛型类实现，代码如下：

```java
@Data
public class Result<T> {

    private int code;
    private String message;
    private T data;

    private Result() {}

    public static <T> Result<T> success() {
        Result<T> result = new Result<>();
        result.setCode(200);
        result.setMessage("success");
        return result;
    }

    public static <T> Result<T> success(T data) {
        Result<T> result = new Result<>();
        result.setCode(200);
        result.setMessage("success");
        result.setData(data);
        return result;
    }

    public static <T> Result<T> error(String message) {
        Result<T> result = new Result<>();
        result.setCode(500);
        result.setMessage(message);
        return result;
    }

    public static <T> Result<T> error(int code, String message) {
        Result<T> result = new Result<>();
        result.setCode(code);
        result.setMessage(message);
        return result;
    }
}
```

`Result`类包含三个字段：`code`表示响应状态码（200成功、500服务器错误、401未认证等）、`message`表示响应消息、`data`为泛型字段承载实际业务数据。该类通过私有构造函数和静态工厂方法的设计模式，确保只能通过`success()`或`error()`创建实例。成功响应提供两个重载：无参版本用于不返回数据的操作（如删除、更新），带参版本用于返回业务数据。错误响应同样提供两个重载：仅消息版本默认使用500状态码，带状态码版本用于401、403等特定场景。所有控制器方法均返回`Result`类型，实现了前后端数据交互格式的标准化。

## 5.2 宠物信息管理模块实现

宠物信息管理模块是平台核心业务模块之一，实现宠物信息的增删改查、分类管理等功能。模块基于MyBatis-Plus的`LambdaQueryWrapper`构建类型安全的查询条件，结合分页插件实现高效的数据检索。

### 5.2.1 宠物列表查询实现

宠物列表查询由`PetController`中的`list`方法提供接口入口：

```java
@GetMapping("/list")
public Result<Page<Pet>> list(@RequestParam(defaultValue = "1") int page,
                              @RequestParam(defaultValue = "10") int size,
                              @RequestParam(required = false) Long categoryId,
                              @RequestParam(required = false) String keyword) {
    return Result.success(petService.listPets(page, size, categoryId, keyword));
}
```

该接口支持四个查询参数：`page`当前页码（默认1）、`size`每页条数（默认10）、`categoryId`分类ID（可选）、`keyword`搜索关键词（可选）。控制器通过`Result.success()`封装分页结果返回。

在服务层，分页查询通过`LambdaQueryWrapper`动态构建条件：`categoryId`不为空时添加等值过滤，`keyword`不为空时在名称和品种字段上添加`LIKE`模糊匹配。查询结果按创建时间降序排列，确保新添加的宠物优先展示。分页由MyBatis-Plus的`Page`对象实现，自动执行`COUNT`查询获取总数，并在SQL中追加`LIMIT`/`OFFSET`完成物理分页，避免一次性加载大量数据带来的性能问题。

## 5.4 商品与订单模块实现

商品与订单模块是平台电商功能的核心，实现商品管理、购物车和订单创建查询等功能。订单创建过程涉及商品校验、金额计算和库存扣减等复杂业务逻辑，需要保证数据一致性。

### 5.4.1 商品管理实现

商品管理同样采用标准CRUD模式。商品实体包含名称、价格、库存、销量、图片、描述等字段，列表支持分页查询与关键词搜索，管理员可在后台进行新增、编辑和删除。商品详情页展示图片轮播、价格、库存状态和商品描述，用户可加入购物车或直接下单。

### 5.4.2 订单创建实现

订单创建是模块中最复杂的操作，由`OrderServiceImpl`中的`create`方法实现，并使用`@Transactional`确保事务一致性：

```java
@Override
@Transactional
public Order create(Long userId, List<OrderItem> items, String address, String phone, String remark) {
    if (items == null || items.isEmpty()) {
        throw new BusinessException("订单商品不能为空");
    }

    BigDecimal totalAmount = BigDecimal.ZERO;
    for (OrderItem item : items) {
        Product product = productMapper.selectById(item.getProductId());
        if (product == null) {
            throw new BusinessException("商品不存在: " + item.getProductId());
        }
        if (product.getStock() < item.getQuantity()) {
            throw new BusinessException("商品库存不足: " + product.getName());
        }
        item.setProductName(product.getName());
        item.setProductImage(product.getImage());
        item.setPrice(product.getPrice());

        totalAmount = totalAmount.add(product.getPrice().multiply(BigDecimal.valueOf(item.getQuantity())));

        product.setStock(product.getStock() - item.getQuantity());
        product.setSales(product.getSales() + item.getQuantity());
        productMapper.updateById(product);
    }

    Order order = new Order();
    order.setOrderNo(generateOrderNo());
    order.setUserId(userId);
    order.setTotalAmount(totalAmount);
    order.setStatus(0);
    order.setAddress(address);
    order.setPhone(phone);
    order.setRemark(remark);
    order.setCreateTime(LocalDateTime.now());
    order.setUpdateTime(LocalDateTime.now());
    orderMapper.insert(order);

    for (OrderItem item : items) {
        item.setOrderId(order.getId());
        orderItemMapper.insert(item);
    }

    return order;
}
```

订单创建方法的执行流程分为五个关键步骤：（1）校验订单商品列表非空；（2）遍历所有订单项，逐一进行商品存在性和库存校验，通过后将商品名称、图片和单价快照至订单项，确保商品信息后续变化不影响历史订单；（3）使用`BigDecimal`精确累加订单总金额，避免浮点精度丢失；（4）生成唯一订单号（前缀"PK"+时间戳+四位随机数）并创建订单主记录；（5）将所有订单项关联订单主记录并逐条写入数据库。

整个方法通过`@Transactional`实现声明式事务管理，任何步骤失败（如商品不存在、库存不足）都会回滚已执行的数据库操作（包括已扣减的库存与已增加的销量），保证数据一致性。库存扣减采用先查询后更新的方式并由事务保护，避免超卖问题。

## 5.5 社区论坛模块实现

社区论坛模块为用户提供交流互动的平台，支持帖子的发布、浏览、搜索以及评论功能。该模块的关键在于需关联查询用户信息和评论统计数据以丰富展示内容。

### 5.5.1 帖子列表查询实现

帖子列表查询由`PostServiceImpl`中的`listPosts`方法实现。该方法不仅查询帖子基本信息，还关联查询发帖用户的昵称与头像，并统计每篇帖子的评论数量。核心代码如下：

```java
@Override
public IPage<PostVO> listPosts(int page, int size, String keyword) {
    Page<Post> postPage = postMapper.selectPage(pageParam, wrapper);

    List<Long> userIds = postPage.getRecords().stream()
            .map(Post::getUserId).distinct().collect(Collectors.toList());
    Map<Long, User> userMap = userIds.isEmpty() ? Collections.emptyMap() :
            userMapper.selectBatchIds(userIds).stream()
                    .collect(Collectors.toMap(User::getId, u -> u));

    List<Long> postIds = postPage.getRecords().stream()
            .map(Post::getId).collect(Collectors.toList());
    Map<Long, Integer> commentCountMap = postIds.isEmpty() ? Collections.emptyMap() :
            postIds.stream().collect(Collectors.toMap(
                pid -> pid,
                pid -> Math.toIntExact(commentMapper.selectCount(
                        new LambdaQueryWrapper<Comment>().eq(Comment::getPostId, pid)))
            ));

    IPage<PostVO> voPage = postPage.convert(post -> {
        PostVO vo = new PostVO();
        vo.setId(post.getId());
        vo.setTitle(post.getTitle());
        vo.setContent(post.getContent());
        User user = userMap.get(post.getUserId());
        if (user != null) {
            vo.setUsername(user.getUsername());
            vo.setAvatar(user.getAvatar());
        }
        vo.setCommentCount(commentCountMap.getOrDefault(post.getId(), 0));
        return vo;
    });

    return voPage;
}
```

该方法采用批量查询优化策略以规避N+1查询问题，实现分三阶段：首先执行帖子的分页查询，过滤非隐藏帖子并按置顶优先、时间降序排列；其次从分页结果中提取不重复的用户ID和帖子ID，分别通过`selectBatchIds`批量查询用户信息和逐条统计评论数，将结果映射为以ID为键的Map；最后利用分页对象的`convert`方法将`Post`实体转换为`PostVO`视图对象，在转换过程中从Map中查找并填充用户信息和评论数量。这种方式将原本N+1次数据库查询优化为固定次数（帖子查询+用户批量查询+评论数统计），显著提升了列表页面的加载性能。

### 5.5.2 评论功能实现

评论功能通过`CommentController`实现。用户在帖子详情页提交评论时，系统将内容、帖子ID和用户ID保存至评论表。评论列表与帖子详情页联动展示，按创建时间升序排列，并关联查询评论者的用户名和头像。此外，帖子详情查询时系统自动将`view_count`加1以统计浏览量。

### 5.6.1 数据统计实现

数据统计功能由`StatsController`中的`dashboard`方法实现，为管理员提供系统运营核心指标：

```java
@GetMapping("/dashboard")
public Result<Map<String, Object>> dashboard() {
    Map<String, Object> stats = new HashMap<>();
    stats.put("userCount", userMapper.selectCount(null));
    stats.put("petCount", petMapper.selectCount(null));
    stats.put("orderCount", orderMapper.selectCount(null));
    stats.put("appointmentCount", appointmentMapper.selectCount(null));
    stats.put("postCount", postMapper.selectCount(null));

    LocalDateTime todayStart = LocalDate.now().atStartOfDay();
    LocalDateTime todayEnd = LocalDate.now().atTime(LocalTime.MAX);
    stats.put("todayOrderCount", orderMapper.selectCount(
            new LambdaQueryWrapper<Order>()
                    .between(Order::getCreateTime, todayStart, todayEnd)
    ));

    return Result.success(stats);
}
```

该方法通过各实体的Mapper接口调用`selectCount(null)`（参数为null表示无条件统计）获取用户、宠物、订单、预约、帖子的总数；再以当天零点为起始、23:59:59为结束构建时间区间，通过`between`条件统计当日新增订单数。所有统计数据封装为Map返回，前端仪表盘据此渲染统计卡片和图表。

## 5.7 前端页面实现

### 5.7.1 前端项目配置

前端项目的开发环境配置在`vue.config.js`中完成：

```javascript
const { defineConfig } = require('@vue/cli-service')

  transpileDependencies: true,
  devServer: {
```

配置指定前端开发服务器运行在8080端口，通过`proxy`代理解决开发环境下的跨域问题：当浏览器请求以`/api`开头的路径时，开发服务器自动将请求转发至`http://localhost:8088`，并设置`changeOrigin: true`修改Origin字段绕过浏览器的同源策略。生产环境通过Nginx反向代理实现同样的效果。

### 5.7.2 Axios请求封装

HTTP请求库Axios通过拦截器实现了Token自动注入与错误统一处理，关键代码如下：

```javascript
const service = axios.create({
  baseURL: 'http://localhost:8088/api',
  timeout: 10000
})

service.interceptors.request.use(
  config => {
    const token = localStorage.getItem('token')
    if (token) {
      config.headers['Authorization'] = 'Bearer ' + token
    }
    return config
  },
  error => {
    return Promise.reject(error)
  }
)

service.interceptors.response.use(
  response => {
    const res = response.data
    if (res.code && res.code !== 200) {
      Message({
        message: res.message || '请求失败',
        type: 'error',
        duration: 3000
      })
      return Promise.reject(new Error(res.message || '请求失败'))
    }
    return res
  },
  error => {
    if (error.response) {
      const status = error.response.status
      if (status === 401) {
        Message({
          message: '登录已过期，请重新登录',
          type: 'warning',
          duration: 2000
        })
        localStorage.removeItem('token')
        localStorage.removeItem('userInfo')
        router.push('/login')
      } else if (status === 403) {
        Message({ message: '没有权限访问该资源', type: 'error', duration: 3000 })
      } else if (status === 500) {
        Message({ message: '服务器内部错误', type: 'error', duration: 3000 })
      }
    }
    return Promise.reject(error)
  }
)
```

请求拦截器从`localStorage`读取JWT令牌并以Bearer Token格式自动注入到请求头。响应拦截器实现两层错误处理：第一层检查业务状态码，当`code`非200时使用Element UI的`Message`弹出错误提示；第二层处理HTTP状态码异常——401自动清除本地令牌和用户信息并跳转登录页，403和500分别提示无权限和服务器错误。该文件还封装了`get`、`post`、`put`、`del`四个便捷方法，简化各业务模块的接口调用。

### 5.7.3 路由配置

前端路由通过Vue Router实现，分为前台和后台两组配置，并通过全局前置守卫实现登录拦截与权限控制：

```javascript
router.beforeEach((to, from, next) => {
  const token = localStorage.getItem('token')
  if (to.path.startsWith('/admin') && !token) {
    next('/login')
  } else if ((to.path === '/login' || to.path === '/register') && token) {
    next('/')
  } else {
    next()
  }
})
```

路由守卫实现两条核心规则：第一，访问以`/admin`开头的后台页面时若无令牌，自动重定向至登录页，防止未授权访问；第二，已登录用户访问登录或注册页面时自动跳转到首页，避免重复登录。前台页面（宠物浏览、社区论坛等）对登录态不做强制要求，仅在发帖、下单、预约等需要身份的操作时才要求登录。

路由采用懒加载方式（动态`import()`语法），每个页面组件在首次访问时才加载对应的JavaScript文件，有效减小初始包体积、提升首屏速度。前台路由挂载在`FrontLayout`布局组件下，后台路由挂载在`AdminLayout`布局组件下，形成清晰的页面层级结构。

### 5.7.4 Vuex状态管理

用户状态管理通过Vuex模块化机制实现，`user.js`模块负责管理登录状态、令牌和用户信息：

```javascript
const state = {
  token: localStorage.getItem('token') || '',
  userInfo: JSON.parse(localStorage.getItem('userInfo') || '{}')
}

const mutations = {
  SET_TOKEN(state, token) {
    state.token = token
    localStorage.setItem('token', token)
  },
  SET_USER_INFO(state, userInfo) {
    state.userInfo = userInfo
    localStorage.setItem('userInfo', JSON.stringify(userInfo))
  },
  CLEAR_USER(state) {
    state.token = ''
    state.userInfo = {}
    localStorage.removeItem('token')
    localStorage.removeItem('userInfo')
  }
}

const actions = {
  login({ commit }, loginForm) {
    return new Promise((resolve, reject) => {
      userApi.login(loginForm).then(res => {
        commit('SET_TOKEN', res.data.token)
        commit('SET_USER_INFO', res.data.user)
        resolve(res)
      }).catch(err => {
        reject(err)
      })
    })
  },
  logout({ commit }) {
    commit('CLEAR_USER')
  }
}
```

该模块的state从`localStorage`初始化，确保页面刷新后登录状态不丢失。`login`动作调用后端登录接口成功后，分别通过`SET_TOKEN`和`SET_USER_INFO`两个mutation将令牌和用户信息同步至Vuex与`localStorage`，实现内存与本地存储的双重持久化；`logout`动作通过`CLEAR_USER`清除全部用户数据。这种设计保证了登录态的持久性与一致性：Vuex提供组件间的响应式共享，localStorage支撑跨页面刷新的状态恢复。

### 5.7.5 前台布局实现
